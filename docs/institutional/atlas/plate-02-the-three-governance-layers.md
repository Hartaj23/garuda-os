# Responsibilities

## Layer 1 — Institutional Governance

Institutional Governance defines **what is legitimate**.

It answers questions such as:

- Should the institution evolve?
- Is constitutional change required?
- What authority exists?
- Which governance cycle should begin?
- Should a constitution be ratified?

Typical artifacts include:

- GAR Constitutions
- Founder Decisions
- Governance Cycle records
- Ratifications
- Constitutional Baseline
- Institutional HOLD

This layer determines institutional legitimacy.
It does not design systems.

## Layer 2 — Architectural Governance

Architectural Governance defines **how constitutional intent is interpreted**.

It answers questions such as:

- How should constitutional principles become architecture?
- What boundaries should exist?
- How should foundations relate to one another?
- How should architecture preserve constitutional intent?

Typical artifacts include:

- ADRs
- Architectural Reviews
- Foundation Designs

Architecture interprets constitutional authority.
It does not create constitutional authority.

## Layer 3 — Engineering Governance

Engineering Governance defines **how architecture becomes implementation**.

It answers questions such as:

- What work is authorized?
- Which sprint should execute?
- What tests are required?
- When is implementation complete?
- When is release appropriate?

Typical artifacts include:

- Sprint Specifications
- Mission Plans
- Source Code
- Tests
- Releases

Engineering implements approved architecture.
It does not redefine architecture.

---

# Authority Flow

Authority always flows in one direction.

```text
Institution
        ↓
Architecture
        ↓
Engineering
```

Authority never flows upward.

Engineering cannot redefine architecture.
Architecture cannot amend constitutions.
Institutional Governance alone determines constitutional legitimacy.

---

# Separation of Responsibilities

| Layer | Primary Question | Creates Authority? |
| --- | --- | --- |
| Institutional Governance | What is legitimate? | Yes |
| Architectural Governance | How should legitimacy be interpreted? | No |
| Engineering Governance | How should architecture be implemented? | No |

This separation allows each discipline to excel without assuming responsibilities that belong elsewhere.

---

# Why This Matters

Many software projects combine governance, architecture, and engineering into a single continuous process.

Project Garuda deliberately separates them.

This separation provides:

- explicit institutional authority,
- traceable decision making,
- architectural fidelity,
- disciplined implementation,
- long-term institutional continuity.

Every meaningful act of creation begins with governance, passes through architecture, and concludes in engineering.

No stage is skipped.
No authority is inferred.

---

# Key Insight

The three governance layers are not organizational departments.
They are institutional responsibilities.

Their purpose is not to slow progress, but to ensure that every meaningful act of creation is supported by explicit authority, faithful interpretation, and disciplined implementation.

Together, they form the structural framework through which Project Garuda governs its own evolution.
